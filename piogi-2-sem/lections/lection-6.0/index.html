Обработка ошибок и UX

Введение

"Error: Object reference not set to an instance of an object."

Такое сообщение понятно разработчику (и то не всегда). А пользователю? Он просто закроет приложение и никогда не вернётся.

Обработка ошибок — это не просто технический аспект. Это часть пользовательского опыта (UX).

Философия ошибок

Fail Fast vs Fail Safe

Fail Fast — падай быстро и громко: - Ошибка обнаруживается сразу - Разработчик быстро находит проблему - Лучше для этапа разработки

public void ProcessPayment(decimal amount)
{
    if (amount <= 0)
        throw new ArgumentException("Amount must be positive");
    // ...
}

Fail Safe — продолжай работать, если возможно: - Система остаётся работоспособной - Пользователь может продолжить - Лучше для production

public async Task LoadMoviesAsync()
{
    try
    {
        var movies = await _api.GetMoviesAsync();
        DisplayMovies(movies);
    }
    catch (HttpRequestException)
    {
        DisplayCachedMovies();  // Показываем кеш вместо краша
        ShowOfflineWarning();
    }
}

Когда что использовать

Ситуация Подход
Ошибка программиста (баг) Fail Fast
Внешняя проблема (сеть, сервер) Fail Safe
Неверный ввод пользователя Валидация + сообщение
Критическая ошибка Fail Fast + логирование

Try-Catch стратегии

Антипаттерн: Поймать и проглотить

// ПЛОХО: Ошибка скрыта, проблема не видна
try
{
    await LoadDataAsync();
}
catch (Exception)
{
    // Ничего не делаем
}

Антипаттерн: Поймать всё

// ПЛОХО: Ловим слишком много
try
{
    var x = int.Parse(input);
    await SaveAsync(x);
}
catch (Exception ex)
{
    // FormatException и NetworkException обрабатываются одинаково?
    MessageBox.Show(ex.Message);
}

Правильный подход: Специфичные обработчики

try
{
    var movies = await _api.GetMoviesAsync();
    MoviesListBox.ItemsSource = movies;
}
catch (HttpRequestException ex)
{
    // Проблема сети
    ShowNetworkError("Проверьте подключение к интернету");
    LogError(ex);
}
catch (JsonException ex)
{
    // Сервер вернул невалидный JSON
    ShowError("Ошибка данных. Попробуйте позже.");
    LogError(ex);
}
catch (TaskCanceledException)
{
    // Запрос отменён — это нормально
}

Ошибки и пользовательский опыт

Плохие сообщения об ошибках

❌ "Error 500: Internal Server Error"
❌ "NullReferenceException at line 42"
❌ "Something went wrong"
❌ "Ошибка"

Хорошие сообщения об ошибках

✅ "Не удалось загрузить список фильмов. Проверьте подключение к интернету."
✅ "Место уже занято. Пожалуйста, выберите другое."
✅ "Сеанс начнётся через 5 минут. Бронирование закрыто."

Компоненты хорошего сообщения

  1. Что случилось — описание проблемы
  2. Почему — причина (если известна)
  3. Что делать — действие для исправления

│  ⚠️ Не удалось забронировать место          │  ← Что случилось

│  другим пользователем минуту назад.         │

Индикаторы загрузки

Зачем нужны

Без индикатора пользователь не понимает: - Приложение работает или зависло? - Нужно ждать или нажать ещё раз? - Сколько ещё ждать?

Типы индикаторов

Indeterminate (неопределённый) — когда не знаем, сколько ждать: 

<ProgressBar IsIndeterminate="True"/>

Determinate (определённый) — когда знаем прогресс: 

<ProgressBar Value="{Binding Progress}" Maximum="100"/>

Skeleton (скелетон) — заглушки вместо контента: 

<Border Background="#E0E0E0" Height="200" CornerRadius="8"/>

Паттерн загрузки

private async void LoadMovies_Click(object sender, RoutedEventArgs e)
{
    // 1. Показать индикатор
    LoadingIndicator.Visibility = Visibility.Visible;
    MoviesListBox.Visibility = Visibility.Collapsed;
    ErrorPanel.Visibility = Visibility.Collapsed;

    // 2. Заблокировать кнопку (предотвращает двойной клик)
    LoadButton.IsEnabled = false;

    try
    {
        // 3. Выполнить операцию
        var movies = await _api.GetMoviesAsync();

        // 4. Показать результат
        MoviesListBox.ItemsSource = movies;
        MoviesListBox.Visibility = Visibility.Visible;
    }
    catch (Exception ex)
    {
        // 5. Показать ошибку
        ErrorMessage.Text = "Не удалось загрузить фильмы";
        ErrorPanel.Visibility = Visibility.Visible;
    }
    finally
    {
        // 6. Скрыть индикатор, разблокировать кнопку
        LoadingIndicator.Visibility = Visibility.Collapsed;
        LoadButton.IsEnabled = true;
    }
}

Retry Patterns

Простой retry

public async Task<T> RetryAsync<T>(Func<Task<T>> action, int maxRetries = 3)
{
    for (int i = 0; i < maxRetries; i++)
    {
        try
        {
            return await action();
        }
        catch (HttpRequestException) when (i < maxRetries - 1)
        {
            await Task.Delay(1000);  // Ждём перед повтором
        }
    }
    throw new Exception("Max retries exceeded");
}

// Использование
var movies = await RetryAsync(() => _api.GetMoviesAsync());

Exponential Backoff

Каждый повтор ждём дольше:

public async Task<T> RetryWithBackoffAsync<T>(Func<Task<T>> action, int maxRetries = 5)
{
    for (int i = 0; i < maxRetries; i++)
    {
        try
        {
            return await action();
        }
        catch (HttpRequestException) when (i < maxRetries - 1)
        {
            // 1s, 2s, 4s, 8s, 16s...
            var delay = TimeSpan.FromSeconds(Math.Pow(2, i));
            await Task.Delay(delay);
        }
    }
    throw new Exception("Max retries exceeded");
}

Circuit Breaker

"Выключатель" — если сервис постоянно падает, перестаём пытаться:

public class CircuitBreaker
{
    private int _failureCount = 0;
    private DateTime _lastFailure;
    private const int Threshold = 5;
    private readonly TimeSpan _timeout = TimeSpan.FromMinutes(1);

    public async Task<T> ExecuteAsync<T>(Func<Task<T>> action)
    {
        // Если "выключатель" открыт — сразу возвращаем ошибку
        if (_failureCount >= Threshold && DateTime.Now - _lastFailure < _timeout)
        {
            throw new CircuitBreakerOpenException("Service unavailable, please try later");
        }

        try
        {
            var result = await action();
            _failureCount = 0;  // Сброс при успехе
            return result;
        }
        catch
        {
            _failureCount++;
            _lastFailure = DateTime.Now;
            throw;
        }
    }
}

Offline-first

Концепция

Offline-first — приложение должно работать (частично) без интернета:

  1. Показываем кешированные данные
  2. Сохраняем действия в очередь
  3. Синхронизируем при восстановлении связи

Простой кеш

public class CachedApiService
{
    private readonly ApiService _api;
    private List<Movie> _cachedMovies;

    public async Task<List<Movie>> GetMoviesAsync()
    {
        try
        {
            _cachedMovies = await _api.GetMoviesAsync();
            SaveToLocalCache(_cachedMovies);
            return _cachedMovies;
        }
        catch (HttpRequestException)
        {
            // Нет сети — возвращаем кеш
            if (_cachedMovies != null)
                return _cachedMovies;

            // Пробуем загрузить из файла
            _cachedMovies = LoadFromLocalCache();
            if (_cachedMovies != null)
                return _cachedMovies;

            throw;  // Совсем нечего показать
        }
    }
}

Graceful Degradation

Концепция

Graceful Degradation — при проблемах функциональность снижается плавно, а не ломается полностью.

Примеры

Проблема: не загрузились постеры - ❌ Падение приложения - ✅ Показываем placeholder-изображение

Проблема: не загрузился список сеансов - ❌ Пустой экран - ✅ Показываем информацию о фильме + сообщение "Сеансы временно недоступны"

Проблема: не работает поиск - ❌ Поиск не отвечает - ✅ Показываем весь список + сообщение "Поиск временно недоступен"

Реализация

private async Task LoadMovieDetailsAsync(int movieId)
{
    // Основные данные — обязательны
    var movie = await _api.GetMovieAsync(movieId);
    DisplayMovieInfo(movie);

    // Сеансы — опционально
    try
    {
        var sessions = await _api.GetSessionsAsync(movieId);
        DisplaySessions(sessions);
    }
    catch (Exception ex)
    {
        DisplaySessionsUnavailable();
        LogWarning(ex);
    }

    // Отзывы — опционально
    try
    {
        var reviews = await _api.GetReviewsAsync(movieId);
        DisplayReviews(reviews);
    }
    catch
    {
        // Молча скрываем секцию отзывов
        ReviewsSection.Visibility = Visibility.Collapsed;
    }
}

Связь с лабораторной работой

Лаба 6: Обработка ошибок

В лабораторной работе вы:

  1. Обработаете разные HTTP-коды: 

    switch (response.StatusCode)
{
    case HttpStatusCode.OK:
        // Успех
        break;
    case HttpStatusCode.Conflict:
        // Место занято
        break;
    case HttpStatusCode.BadRequest:
        // Ошибка валидации
        break;
}

  2. Покажете понятные сообщения пользователю

  3. Реализуете индикатор загрузки: 

    BuyButton.Content = "Обработка...";
BuyButton.IsEnabled = false;

Заключение

Обработка ошибок — это не "потом доделаем", это основа хорошего UX:

  • Ловите ошибки там, где можете с ними что-то сделать
  • Показывайте понятные сообщения, а не технический мусор
  • Используйте индикаторы загрузки
  • Приложение должно деградировать gracefully, а не падать

Пользователь простит медленную загрузку. Но не простит необъяснимый краш.
